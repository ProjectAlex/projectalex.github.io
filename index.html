Literature Survey (Contd.)

  • Multiplayer Game capacity can be increased using the optimisation methods suggested by Hanghang Qi and David Malone, University of Ireland in "802.11 Wireless LAN Multiplayer Game Capacity and Optimization". In 802.11e wireless networks, with proper configuration of TXOP at the game server and access points, the network can be improved to support upto 15 players [A TXOP(Transmit Opportunity) is a bounded time interval during which a station can send as many frames as possible].

    • The Game

    • The game is an adaptation of "Snakes and Ladders" combined with some features of "Scene It"
    • The game can be played by as many players as we want. For now we want to limit it to the number of people in our group, that is 5
    • Its a turn based game. You advance only when its your turn.
    • A question appears on the screen, and all the participants are supposed to answer the question
    • Only the person whose turn it was advances, opponents get to limit the number of positions the player can advance.

    The Game (Contd.)

    • The question will be a combination of pictures, videos, audio clips in various topics to exploit the fact that the game is digital.
    • The first person to reach the finish point is declared winner.
    • Each square has a different genre of question assigned to it. So the square you are in determines the question you will be asked in the next turn.
    • The system has a database of questions, and we train the system by answering the questions, enabling it to identify the difficulty level.

    For eg

    • For a 5 player game if 3 people answer the question quicker than the player playing his hand he moves 2 spaces ahead, or if he is the first to answer he moves 5 spaces ahead.
    • This keeps all the players engrossed as they are constantly interacting with the game.

    Design

    • The board appears on the screen.
    • The current player in turn is displayed on the left corner at the top, whereas points of the opponents appear on the right corner.
    • The question appears down along with its options, and this appears on the devices as well.
    • The options can be selected only on the phone

    Implementation

    Implementation

    • The Raspberry Pi runs a Server running Java
    • We have a centralized corpus of questions running on a VPS.
    • The pi handles displaying the Board UI to the Screen
    • The pi also fetches questions from the VPS.
    • The pi sends questions, options and scores to the mobile through sockets.

    Implementation (Contd.)

    • The client would be a native android app. (Could be extended to other platforms later).
    • The android app displays the questions, options, and the user needs to pick the answer there.
    • The communication between the Raspberry pi and the android app would happen through sockets.

    Existing Solutions and Problems

    • Google's Chromecast is not opensource and thus does not give enough flexibility.
    • PiCast runs on a raspberry Pi and allows devices to content stream over WiFi. It is priced similar to the chromecast making it economically feasible. Devices communicate with PiCast using an API. It supports only Videos, Audio, Images like Chromecast
    • Another device we considered was Roku Stick which uses DLNA to stream content and can be plugged into any HDMI compatible TV.

    Challenges we face

    • Speed - The connection speed may cause problems during gameplay.
    • Maximum number of players the server can handle will be a limiting factor.
    • Should be interconnected using WiFi
    • Fluctuations in speed and connections

    How do we plan to tackle
    the challenges

    • The game must be tweaked according to the connection speed that is achievable.
    • The upper limit for number of players should be set based on the servers capability and the game that is chosen
    • The app must be able to work under a wobbly connection allowing players to reconnect and continue and not get kicked out
